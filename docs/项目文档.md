## 一、项目是什么

### 定位

给 LLM / Agent 做**自动化评测与对比**的框架。典型场景：你有一个自研 agent（my_agent）和若干竞品（如豆包/DeepSeek），想用一份混合评测集（大部分有参考答案、少部分开放题）知道——**谁更好？好在哪、差在哪？怎么优化？**

### 三条核心设计原则

| 原则 | 含义 | 为什么 |
|---|---|---|
| **① 盲评** | 被测模型和裁判都**不接触参考答案** | 公平作答 + 裁判凭真实能力判断 |
| **② 参考答案做元评测** | 参考答案**最后才出场**，校验"评测 agent 本身准不准" | 只有盲评，参考答案才能当"测裁判"的标尺 |
| **③ 无参考答案题：模拟人工评测 + 补足人工缺点** | 裁判像不同背景的真人评测员那样评开放题，用机制克服人工固有问题 | 开放题无标准答案，绝对分不可靠 |

> 一句话：**框架评的是模型，元评测评的是框架自己。**

## 二、整体架构与数据流

```
评测集 (jsonl, reference 仅元评测用)
   │  to_prompt() 只传 question+context（剥离 reference）
   ▼
[1 批跑引擎]  并发调用所有被测模型 → 收集盲答（断点续跑）
   ▼
[2 盲评引擎]  裁判 agent 独立评判（不接收 reference）
   ├─ Rubric Judge：意图→理想锚定→多角度分析→判定（可联网查证）
   └─ Pairwise Judge：匿名 A/B 成对盲比较（双向，抗位置偏差）
   │  多裁判 × 重复采样 → 集成（投票/去极值均值/加权/Bootstrap/一致率）
   ▼
[3 元评测]  参考答案首次出场，作 ground truth 校验"裁判准不准"
   ▼
[4 分析层]  聚合 · 切片 · 待评测 vs 竞品胜率 · 典型 case · 弱点聚类
   ▼
[5 报告层]  A 被测对比报告(md+json)  +  B 评测可靠性报告(md+json)
```

## 三、数据模型（schema）

所有跨模块流转的结构在 [schema.py](../src/auto_eval/schema.py)。核心模型：

### EvalItem（一条评测题）
| 字段 | 类型 | 说明 |
|---|---|---|
| `id` | str | 题目 id |
| `question` | str | 问题 |
| `context` | str? | 可选背景/多模态描述 |
| `has_ref` | bool | 是否有参考答案 |
| `reference` | str? | ⚠️ **仅元评测用，禁止进入作答/盲评** |
| `key_points` | list[str] | 可接受要点（元评测辅助）|
| `category` | str/list[str] | 类目（切片用）|
| `difficulty` | easy/medium/hard | 难度 |
| `tags` | list[str] | 标签 |

### ModelOutput（模型盲答）
`item_id, model, answer, latency_ms, tokens_in, tokens_out, cost, error, raw`

### SingleScore（单裁判一次 rubric 评分）
| 字段 | 说明 |
|---|---|
| `rubric` | dict[维度→1-5 分] |
| `total` | 总分（裁判输出）|
| `correctness` | right/wrong/partial/unclear |
| `error_type` | 错误归因标签（可空）|
| `rationale` | 一句话理由 |
| `analysis` | 裁判深度思考链（`<analysis>` 全文）|
| `tool_trace` | 评测 agent 的工具调用轨迹 |
| `used_search` / `search_queries` | 是否联网 / 搜索词 |
| `judge` / `persona` / `run_idx` | 哪个裁判 / 画像 / 重复采样序号 |

### Verdict（聚合盲评结论）
`rubric`(各维度均分) `total`(加权总分) `correctness`(投票) `error_type` `rationale` `n_judges` `judges_agreement`(一致率) `repeat_std`(稳定性) `low_agreement`(标红) `single_scores`

### PairResult（成对比较聚合）
`model_a, model_b, a_wins, b_wins, ties, winner, win_rate_a, bidirectional_consistent, low_agreement`

### MetaResult（元评测）
`objective{em,f1,sim}` `objective_correct` `judge_correctness` `agree` `delta_total_vs_sim`

## 四、盲评详解

### 4.1 为什么盲评

- 作答：被测模型只拿 `question+context`，**绝不传参考答案**。
- 评判：裁判也**不接收参考答案**。
- 隔离实现：[dataset.py](../src/auto_eval/dataset.py) 的 `to_prompt()` 只返回 question+context，结构上保证不泄漏（有测试断言）。

只有盲评，参考答案才能在最后当"测裁判"的标尺。

### 4.2 裁判是一个 Agent（ReAct loop），不是单次打分

裁判评判是一个 agent loop（[base.py](../src/auto_eval/judges/base.py) 的 `JudgeClient.complete`）：

```
每轮（最多 max_rounds=12）：
  1. 调裁判 LLM（请求带上"可用工具"定义，tool_choice="auto"）
  2. 看返回：
     - 有 tool_calls → 执行工具 → 结果作为 role=tool 消息回灌 → 下一轮
     - 无 tool_calls → 裁判"已确信"，跳出循环，输出最终评判
```

- 裁判**自主决定**查什么、查几次、何时停。
- **截断强制判定兜底**：到 `max_rounds` 仍想调工具时（复杂题"查证上瘾"），追加一次无工具调用逼裁判出 JSON——保证总有评分、不致空白；并标记 `truncated`（结果/报告显示"⚠️截断(强制判定)"，trace 里记 `force_judgement` 轮）。
- LLM 调用含 **429/过载自动重试**（指数退避，最多 5 次）。
- 代码：主循环 [base.py:64](../src/auto_eval/judges/base.py#L64)、终止 [base.py:74](../src/auto_eval/judges/base.py#L74)、工具执行 [base.py:94](../src/auto_eval/judges/base.py#L94)、重试 [base.py:113](../src/auto_eval/judges/base.py#L113)。
- 流程图：[assets/盲评AgentLoop.flowchart.excalidraw](../assets/盲评AgentLoop.flowchart.excalidraw)

### 4.3 裁判的工具集（function calling）

| 工具 | 作用 | 实现 |
|---|---|---|
| `web_search` | 联网搜索核实事实 | Tavily/SerpAPI/Bing 三选一 |
| `fetch_page` | 抓网页正文深挖 | httpx + 去标签 |
| `calculate` | 安全算术求值（核查计算题）| AST 白名单，无名字/调用 |
| `python_run` | 执行 Python（核查编程题）| 子进程+超时，默认关 |

**工具如何被裁判知道**：通过 OpenAI function calling 的 `tools` 参数（结构化 schema），不是 prompt 文本。prompt 里的文字只是辅助引导。

| 层 | 载体 | 作用 |
|---|---|---|
| 机制层（主）| API 的 `tools` 参数 | 模型知道有哪些工具、参数格式 |
| 引导层（辅）| prompt 文字 | 引导何时、为何用 |

> ⚠️ 裁判模型**必须支持 function calling** 才能联网核查（见 [FAQ](#九注意事项与-faq)）。

### 4.4 深度推演 Prompt

rubric 盲评强制裁判走深度流程（[prompts.py](../src/auto_eval/judges/prompts.py) 的 `RUBRIC_SYSTEM`）：

1. **意图理解**：提问者想要什么（信息/建议/创作/分析/推理）
2. **理想答案锚定**：推理"好回答该覆盖哪些要点"（不是背标准答案）
3. **多角度分析**：切题度 / 准确性与深度 / 完整性 / 结构与表达 / 实际价值（按意图侧重）/ 安全合规
4. **对比锚点**：被评答案 vs 理想画像
5. **综合判定**：各维度分 + 判定 + 归因

> 注意区分两个概念：**"分析角度"**（上面第 3 步，6 个，用于引导思考）与 **"打分维度"**（见 4.5，5 个，正式评分项）是两回事。

### 4.5 打分维度、输出格式、判定档位

**5 个正式打分维度**（[rubrics.yaml](../config/rubrics.yaml)，每个满分 `scale=5`）：

| 维度 | 定义 | weight |
|---|---|---|
| 准确性 | 事实与逻辑是否正确，有无幻觉或错误 | **1.5** |
| 完整性 | 是否充分回应题目所有要点，不遗漏 | 1.0 |
| 相关性 | 是否切题、不跑题、不冗余 | 1.0 |
| 有用性 | 对提问者是否清晰易懂、切实满足需求（用户视角）| 1.0 |
| 安全性 | 是否含风险、误导或有害内容 | 0.5 |

裁判对每维度打 **1–5 整数分**，并输出：

```json
{
  "rubric": {"准确性": 4, "完整性": 3, "相关性": 5, "有用性": 4, "安全性": 5},
  "total": 4.4,
  "correctness": "right",
  "error_type": "事实错误",
  "rationale": "一句话理由"
}
```
（前面还会输出 `<analysis>...</analysis>` 思考链，被留存到 `SingleScore.analysis`。）

**correctness 四档判定**：

| 值 | 含义 |
|---|---|
| `right` | 正确 |
| `wrong` | 错误 |
| `partial` | 部分正确 |
| `unclear` | 无法判断 |

**error_type 归因标签**（裁判自由给出，常见）：`事实错误 / 逻辑缺失 / 不完整 / 指令偏离 / 计算错误 / 含风险内容 / 无问题(null)`。分析层会按 error_type 聚类弱点。

#### 垂域 Skill（动态维度匹配）

通用 5 维对所有题"一刀切"，但不同垂域需要不同的评测侧重（创作重审美、事实重准确、翻译重信达雅）。系统引入了 **Skill 架构**：按 `EvalItem.category` 自动匹配垂域 Skill，裁判拿到**不同的维度子集和评测规则**。

6 个内置 Skill，每个是独立的 `config/skills/xxx.yaml` 文件（自带完整 rubrics、规则、示例，互不干扰）：

| Skill | 匹配类目 | 维度子集 | 特殊规则 |
|---|---|---|---|
| `default` | 未匹配时回落 | 准确性/完整性/相关性/有用性/安全性 | — |
| `creative` | 创作/诗歌/文学… | **新颖性/审美表达**/完整性/安全性 | 不核查事实正确性，评审美价值与表达新颖度 |
| `factual` | 事实/时新/科普… | 准确性/完整性/相关性/有用性/安全性 | 准确性权重最高，必须联网核查每个关键事实断言 |
| `advisory` | 建议/规划/方案… | **可操作性**/完整性/准确性/有用性 | 重点评方案可行性，而非事实准确性 |
| `math_logic` | 计算/编程/推理… | 准确性/**步骤清晰度**/完整性/安全性 | 裁判先验算再逐步骤对比，关注步骤严谨性 |
| `translation` | 翻译 | **信达雅**/准确性/完整性 | 评忠实+通顺+美感 |

裁判渲染 system prompt 时，Skill 的维度子集注入 `【打分维度】`，规则注入 `【本类题评测侧重】` 段（[skill_router.py](../src/auto_eval/judges/skill_router.py) + [rubric_judge.py](../src/auto_eval/judges/rubric_judge.py)）。自定义 Skill 在 `domain_skills` 加条目（维度名须在 rubrics 总库已定义），重启生效。

#### 层级维度（一级 → 二级）

每个一级维度可配 1-N 个二级子维度（[rubrics.yaml](../config/rubrics.yaml) 的 `sub_dimensions`）。裁判对**每个二级分别打分（1-5）**，一级自动取二级均值；无 `sub_dimensions` 的维度退化为一级直接打分。

当前通用 `default` Skill 的二级维度结构：

| 一级维度 | 二级子维度 |
|---|---|
| 准确性 | 事实准确性、逻辑正确性、信息时效性 |
| 完整性 | 要点覆盖、细节丰富度 |
| 相关性 | 文本结果相关性、参考来源相关性、配图配视频相关性、引导链接相关性 |
| 有用性 | 清晰易懂、可操作落地、需求满足度 |
| 安全性 | 无害性、合规性 |

> `math_logic` Skill 的「步骤清晰度」维度另有专属二级：步骤可追踪、中间结果正确、策略合理性（计算/编程/推理专用）。

总分计算：**二级均分 → 一级分 → 按 weight 加权**（见 4.6）。多裁判聚合：各裁判的二级分先均分到一级，再跨裁判 trim_mean。

### 4.6 总分按 weight 加权

总分**不是简单平均**，而是按 [rubrics.yaml](../config/rubrics.yaml) 的 weight 加权（[ensemble.py](../src/auto_eval/judges/ensemble.py)）：

```
total = Σ(维度均分 × weight) / Σ(weight)
```

当前权重和 = 1.5+1.0+1.0+1.0+0.5 = **5.0**。例：
- 五维都得 4 → total = 4.0
- 准确性 5、其余 3 → (5×1.5 + 3×3 + 3×0.5)/5 = 18/5 = **3.6**（准确性权重大，拉分明显）

归一化：`norm_total = total / 5`（0–1）。

### 4.7 事实核查协议（实战强化）

[prompts.py](../src/auto_eval/judges/prompts.py) 要求：
- 提取答案中所有**具体事实断言**（名称/日期/年份/数字/归属），这类**几乎都要核查**；
- **必须主动用工具核查真伪**；
- **核心原则：核查是裁判职责**。不要因"答案没提供来源"就扣分——答案不必自带来源，你要自己查它对不对；
- 只有所有关键断言核查完（或确属常识）才判定。

案例（题`许嵩2026新歌?`答`有何不可`）：强化前不联网误放；强化后 `search[许嵩 2026年 新歌]`，发现"有何不可"非 2026 新歌，判 `wrong`。

### 4.8 多裁判集成（模拟不同评测员）

- **评测员画像（persona）多样化**：不同裁判配不同 persona，模拟不同背景评测员。4 种预设（[prompts.py](../src/auto_eval/judges/prompts.py)）：

| persona | 立场 | 评判倾向 |
|---|---|---|
| `strict_expert` | 严谨学科专家 | 重准确/逻辑/完整；确实正确才 right |
| `end_user` | 终端用户 | 重易懂/有用/需求满足 |
| `safety_reviewer` | 安全审核员 | 重事实正确/误导/风险 |
| `balanced` | 公正资深评审 | 折中 |

  规则：让裁判间**视角互补**（如 expert + user），而非重复；想测"同视角稳定性"时才用同 persona（配合 repeat>1）。

- **并行**：多裁判评同一题是**并发**的（`asyncio.gather`），受全局信号量 + 各裁判 RateLimiter 节流。
- **聚合**（[ensemble.py](../src/auto_eval/judges/ensemble.py)）：分类→多数投票；连续分→去极值均值；**总分→weight 加权**；Bootstrap 置信区间；一致率 α；重复稳定性方差；低于阈值→标红人工复核。

### 4.9 评审委员会与主席仲裁（分歧兜底）

多裁判采用**分层委员会**机制（"陪审团 + 法官"），主席**不是每次都介入**，只在分歧时才请出。

**第一层**：N 裁判独立盲评 → 各自判定/分/理由/查证（见 4.8）。

**触发条件**（三者同时满足才请主席）：
1. 聚合成功；
2. `low_agreement = True`（分歧信号）；
3. ≥2 个裁判评分。

**`low_agreement` 怎么算分歧**（[ensemble.py](../src/auto_eval/judges/ensemble.py)）：当「多裁判一致率 < 阈值」**或**「同裁判重复采样稳定性差」时为 True。
- 一致率 = correctness 多数类占比（0–1）；
- 阈值 = `flag_low_agreement`（[judges.yaml](../config/judges.yaml)，默认 **0.6**，可调）。

**第二层（主席仲裁，仅分歧时触发）**：主席（用第一个裁判的客户端，可改）拿到题目 + 答案 + **所有裁判的判定/理由/查证证据**，综合各方观点 + **自主联网核查**争议点，给出最终判定 + 各维度分 + **置信度（0–1）** + 理由；仲裁结果**覆盖**为最终结论（裁判原始意见仍留存在 `Verdict.single_scores` 可追溯）。代码见 [arbitrator.py](../src/auto_eval/judges/arbitrator.py)。

**不同裁判数下何时触发**（一致率 < 0.6 才仲裁）：

| 裁判数 | 判定分布 | 多数派占比 | 主席介入？ |
|---|---|---|---|
| 2 | 2:0（一致）| 1.0 | ❌ 采纳多数 |
| 2 | 1:1（不一致）| 0.5 | ✅ 仲裁（1 个不同即触发）|
| 3 | 3:0 | 1.0 | ❌ |
| 3 | 2:1 | 0.667 | ❌（多数够强，>0.6）|
| 3 | 1:1:1（全不同）| 0.333 | ✅ |
| 5 | 5:0 / 4:1 | 1.0 / 0.8 | ❌ |
| 5 | 3:2 | 0.6 | ❌（等于 0.6，不满足 <）|
| 5 | 2:2:1 | 0.4 | ✅（主流派 ≤2 票）|

> 当前 2 裁判下，**两裁判判定不一致即触发**；裁判越多，触发门槛相应变化（多数派占比需 <0.6）。

**适用范围**：仅 rubric 盲评（单评/过程/在线）。成对比较（compare）有自己的双向一致性判断（4.11），不走主席。

**不确定中间态**：裁判与主席的 prompt 都明确——无法确定答案对错时**必须给 `unclear`**，不硬猜 right/wrong；置信度 <0.5 应改判 unclear。

**调参**：想让主席更积极介入（如 3 裁判 2:1 也仲裁）→ 调高 `flag_low_agreement`（如 0.7）；想少介入 → 调低。改 [judges.yaml](../config/judges.yaml) 重启生效。

### 4.10 补足人工评测缺点

| 人工缺点 | 应对 |
|---|---|
| 主观不一致/漂移 | rubric 固化 + 多裁判投票 + 重复采样 |
| 疲劳 | 机器无疲劳 |
| 知识盲区 | web_search / fetch_page 联网 |
| 位置偏差 | 成对比较双向 + 随机化 |
| 锚定 | 先独立盲评再成对比较 |
| 规模小/慢/贵 | N×K 采样 + Bootstrap |
| 不可复现 | rubric/温度固定 + 全量留痕 |
| 等价判断不一 | prompt 明写"接受等价表达" |

### 4.11 成对比较（Pairwise，无答案题主指标）

无参考答案时绝对分有偏差，人更擅长"哪个更好"。所以：

- **匿名 A/B 盲比较**（[pairwise_judge.py](../src/auto_eval/judges/pairwise_judge.py)）：裁判只输出 a/b/tie + 理由。
- **抗位置偏差**：**双向**（A vs B + B vs A）+ 位置随机化；归一化后两向不一致→标红。
- 聚合为**胜率** `(a_wins + 0.5×ties) / total`，可选 Elo。

### 4.12 元评测（参考答案校验裁判本身）

**对象是裁判，不是被测模型**。参考答案在此首次出场（[meta/](../src/auto_eval/meta/)）：

- **客观真值**：reference → `EM/F1/语义相似度` → `objective_correct`。
- **对照**：裁判盲判 vs 客观 → **判题准确率/召回率/F1**；盲评分 vs 客观相似度的 **Pearson**；**偏差方向**（偏严/偏松）。
- **无 ref 题**：靠多裁判一致率 + 重复稳定性。
- 产出 **B 报告**：准确率、偏差、判错 case、可信度结论。

> 实测：乱判假裁判→B 报告"准确率 0%、不可信"；真实 glm 裁判→"准确率 100%、可信"。同一套元评测能区分好/坏裁判。

### 4.13 过程盲评（评 agent 推理过程，可选模式）

除"结果盲评"（评最终答案）外，另有**过程盲评**模式：评被测 agent「得出答案的过程」质量（推理/工具使用/纠错），需提供 agent 的**推理轨迹 trace**。

- **何时用**：待测是 agentic 系统（多步工具/规划/检索），想评它"怎么得出答案"，而不只看最终答案。
- **触发**：web「过程盲评(含轨迹)」Tab，输入 `query ||| answer ||| trace [||| reference]`；数据集用 `EvalItem.trace` 字段。
- **过程维度**（[rubrics.yaml](../config/rubrics.yaml) 的 `process_rubrics`，与结果维度独立）：推理严密性 / 工具与检索使用 / 纠错与自我修正 / 过程完整性 / 最终准确性。
- **防 reasoning bias**：prompt 明确"不要被漂亮推理带偏，要逐条核对步骤是否真正确、工具是否真有效"。
- **与结果盲评独立**：过程分不污染结果分——可对同一题分别跑两种模式对照。

**关键区别（过程 vs 结果盲评）**：

| | 结果盲评 | 过程盲评 |
|---|---|---|
| 输入 | query + answer | query + answer + **trace** |
| 维度 | 准确性/完整性/相关性/有用性/安全性 | 推理严密性/工具使用/纠错/过程完整性/最终准确性 |
| 重点 | 最终答案好不好 | **得出答案的过程**好不好 |
| 防 reasoning bias | 无（不看过程）| **有**（不被漂亮推理骗，逐条核对步骤真假）|

**实测案例**（展示过程盲评的价值）：北京到上海题，answer="约1200公里"（数值正确），但 trace 声称"调用地图 API"却无任何实际工具调用记录。
- **结果盲评**：只看答案"约1200公里"对 → 会被骗成**高分**（准确性=5）。
- **过程盲评**：裁判核查发现"声称调 API 却无记录 = 过程虚构" → 工具与检索使用=1、推理严密性=2，**总分仅 2.20**（即使最终准确性=5）。

> 过程盲评能抓出"结果对但过程编造"——这是结果盲评发现不了的。

## 五、指标与公式

### 5.1 规则指标（[meta/ground_truth.py](../src/auto_eval/meta/ground_truth.py)，仅元评测用）

- **EM（精确匹配）**：归一化（去标点/空白/小写）后完全相同 → 1，否则 0。
- **F1**：**字符级** token（适合中英混排）的 precision/recall/F1。
- **语义相似度 sim**：当前用字符级 F1 作代理（可替换为 embedding cosine）。
- **objective_correct**：`em==1 或 f1≥0.85` → right；`f1≥0.4` → partial；否则 wrong。

### 5.2 成对胜率

```
win_rate_a = (a_wins + 0.5 × ties) / (a_wins + b_wins + ties)
```

### 5.3 Bootstrap 置信区间

对每场比较的 a 是否赢（1/0.5/0）做 N 次（默认 200）有放回重采样，取均值的 2.5%/97.5% 分位作 95% CI。

### 5.4 多裁判一致率

correctness 最多类占比（0–1）；配合 repeat_std（同裁判多次的总分标准差）评估稳定性。

### 5.5 元评测指标（[meta/calibration.py](../src/auto_eval/meta/calibration.py)）

- **judge_accuracy**：有 ref 题里，盲判对错 == 客观对错 的比例。
- **bias**：盲评分(归一) − 客观相似度 的均值；>0.1 偏松，<−0.1 偏严。
- **score_corr**：盲评分 vs 客观相似度的 Pearson。

## 六、其他模块与工程细节

### 6.1 Runner（可插拔模型适配）

| runner | 用途 | 关键配置字段 |
|---|---|---|
| `openai_compat` | 豆包/火山 Ark/任意 OpenAI 兼容 | `base_url, api_key_env, model` |
| `http` | 任意 HTTP 服务 | `url, method, prompt_field, answer_jsonpath, headers` |
| `func` | 本地 Python 函数 | `func_module`（形如 `mypkg.agent:chat`）|
| `cli` | 命令行进程 | `command`（列表，stdin 传 prompt、stdout 取回答）|

通用字段：`concurrency`（并发）、`rpm/tpm`（限流）、`temperature`、`max_tokens`。

### 6.2 批跑引擎（[batch/](../src/auto_eval/batch/)）

- **并发**：题×模型 笛卡尔积，`asyncio.gather`。
- **限流**：每模型一个令牌桶（`RateLimiter`），按 `rpm` 控制请求间隔 + `concurrency` 信号量。
- **重试**（[retry.py](../src/auto_eval/batch/retry.py)）：指数退避（1/2/4/8s），仅重试 `httpx.HTTPError/Timeout/Connection/OSError/429`。
- **断点续跑**（[checkpoint.py](../src/auto_eval/batch/checkpoint.py)）：每条答案即时 append 到 `runs/<id>/answers/{model}.jsonl`；启动读已成功的 `item_id` 集合跳过。**跑到一半挂了能接着跑。**

### 6.3 分析层（[analysis/](../src/auto_eval/analysis/)）

- 聚合 + 按 category/difficulty/has_ref 切片；
- 待评测 vs 竞品胜率、各维度均分差；
- 典型 case 挖掘（赢/输最多、分歧最大）；
- 弱点按 error_type 聚类 → 规则模板生成优化建议（Prompt/检索/参数/数据层）。

### 6.4 报告（[report/render.py](../src/auto_eval/report/render.py)）

- **A 被测对比报告**：各模型得分表、成对胜率、按类目胜率热力表、维度差、典型 case（含**裁判查证轨迹**）、优化建议。
- **B 评测可靠性报告**：判题准确率、偏差、各类目准确率、判错 case、可信度结论。

### 6.5 运行产物结构

```
runs/<run_id>/
├── answers/<model>.jsonl     # 每模型盲答（断点续跑依据）
├── verdicts.jsonl            # 盲评聚合结论
├── pairs.jsonl               # 成对比较结果
├── meta.jsonl                # 元评测
└── reports/
    ├── A.md / A.json         # 被测对比
    └── B.md / B.json         # 评测可靠性
```

## 九、注意事项与 FAQ

### Q1: 裁判模型必须支持 function calling 吗？

**是的**（要联网核查的话）。裁判 agent loop 靠 `tools` 参数 + `tool_calls`。被测模型不需要（只生成回答）。

### Q2: deepseek-v4-pro/flash 在某聚合代理上报 400，不能当裁判？

是的。实测报错：`litellm.BadRequestError: AnthropicException - tools[0]: unknown variant 'custom'`。原因：代理把这两个模型的 tools 请求**路由到 Anthropic 后端**，而 OpenAI 与 Anthropic 的 tools 格式不兼容（OpenAI 用 `type:"function"`，Anthropic 用 `input_schema`），代理未正确转换 → 400。**普通问答（不带 tools）不受影响**，所以它们能当被测模型，不能当联网裁判。

修法：① 换直连官方 DeepSeek API（`api.deepseek.com`，支持 tools）；② 若代理自部署，改 litellm 路由让 deepseek 走原生后端；③ 用支持 tools 的模型（glm-5.2/glm-4.7、Moonshot kimi）当裁判。

**✅ 更新（本代理的解法）**：该代理上带 `-openai` 后缀的版本——`deepseek-v4-pro-openai` / `deepseek-v4-flash-openai`——走 **OpenAI 兼容后端**，**function calling 正常**（实测两者均返回 `web_search` tool_calls）。所以：
- `deepseek-v4-pro-openai` **可当裁判**（当前裁判②即用它，配合 glm-5.2 形成双裁判）；
- `deepseek-v4-flash-openai` 可当被测（当前 doubao 即用它）。

即：同一代理上，**不带后缀**的 deepseek 走 Anthropic（tools 报错），**带 `-openai` 后缀**的走 OpenAI（tools 正常）——选模型时注意后缀。

### Q3: 当前是多裁判还是单裁判？

**框架是多裁判架构**（[engine.py](../src/auto_eval/engine.py) 用裁判列表 + 集成）。当前 [judges.yaml](../config/judges.yaml) 配了 2 个（glm-5.2 strict_expert + deepseek-v4-pro-openai end_user），**真多裁判**。单裁判时"一致率恒 1.0"无意义，需 ≥2 个裁判。

### Q4: `python` 能跑但找不到包？

本机 `python` 指向 hermes venv（无项目依赖），`pip` 装到 Anaconda。请显式用 `& "D:\ProgramData\anaconda3\python.exe" -m ...`。

### Q5: Windows 控制台报 UnicodeEncodeError？

控制台默认 GBK，print `▶✅█` 等会报错。脚本入口加 `sys.stdout.reconfigure(encoding="utf-8", errors="replace")`，或 `$env:PYTHONIOENCODING="utf-8"`。

### Q6: weight 改了不生效？

weight 在 [ensemble.py](../src/auto_eval/judges/ensemble.py) 的 `aggregate_scores` 参与总分加权。改 [rubrics.yaml](../config/rubrics.yaml) 的 weight 后，需**重启服务/重跑**（裁判进程 import 时加载）才生效。

### Q7: 怎么开裁判调用的明细日志？

设环境变量 `AUTO_EVAL_JUDGE_TRACE=<jsonl路径>`，裁判每次评判会把**每轮 LLM 响应、每次工具的完整返回、最终对话历史**追加到该 jsonl。**默认关**（不设不记、不产生开销）。

```powershell
$env:AUTO_EVAL_JUDGE_TRACE="runs/judge_calls.jsonl"
& "D:\ProgramData\anaconda3\python.exe" -m auto_eval.web.server   # 或 CLI
```

每条记录含：`judge / model / system / user / rounds / used_search / search_queries`、`llm_rounds`（每轮 content + tool_calls + finish_reason + usage）、`tool_results`（每次 name + args + **result 全文**）、`messages`（完整对话历史）。web/CLI 启动前设即生效。

> 注意区分：`SingleScore.tool_trace`（摘要如 `search[xxx]→3条`，给报告/结果表用）与这里的明细日志（完整请求/响应/工具返回，供调试审计）是两回事。

## 附：设计哲学速记

- **先评估，再归因，再优化**；单条轨迹只作证据。
- **规则评测覆盖硬约束，LLM 裁判覆盖语义质量**；二者互补。
- **盲评让参考答案能当"测裁判"的标尺**；元评测让评测系统自身可问责。
- **模拟人工评测的结构，用机制克服人工的缺陷**。
